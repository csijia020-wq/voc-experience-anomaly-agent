## 一、知识分区

### 分区1：业务知识库

- **分区名称**：业务知识库（Business Knowledge）
- **内容描述**：包含各业务线的业务背景、数据集映射关系、口径说明等
- **数据来源**：
  - 数据集ID映射表（业务名称 → 数据集ID）
  - 业务口径文档（万服计算口径、维度定义）
  - 业务背景介绍（各业务线简介）
- **更新频率**：业务线变更时更新

### 分区2：分析知识库

- **分区名称**：分析知识库（Analysis Knowledge）
- **内容描述**：包含异动判断规则、分析维度定义、告警类型说明等
- **数据来源**：
  - 异动判断规则文档（YoY阈值、服务量变化占比阈值、极端值定义）
  - 分析维度说明（城市等级、品类、事件类别、进线渠道、战区、FAQ）
  - 告警类型定义（new_category、extreme_value、near_zero等）
- **更新频率**：分析规则变更时更新

### 分区3：操作知识库

- **分区名称**：操作知识库（Operation Knowledge）
- **内容描述**：包含Agent使用指南、FAQ、常见问题解答等
- **数据来源**：
  - Agent使用指南（如何输入命令、管理任务等）
  - 常见问题解答（数据查询失败怎么办、报告生成失败怎么办等）
  - 最佳实践案例（优秀报告示例、分析结论撰写模板）
- **更新频率**：用户反馈问题后持续更新

### 分区4：历史case库

- **分区名称**：历史Case库（Historical Cases）
- **内容描述**：包含历史分析报告、Badcase记录、优秀分析案例等
- **数据来源**：
  - 历史分析报告（已生成的周报数据）
  - Badcase记录（分析错误的案例及修正方案）
  - 优秀分析案例（分析结论撰写优秀的案例）
- **更新频率**：每次分析后沉淀新案例

## 二、字段设计

### 知识条目 Schema

```json
{
  "id": "唯一标识（UUID）",
  "title": "标题",
  "content": "内容",
  "category": "分类",
  "subcategory": "子分类",
  "tags": ["标签1", "标签2"],
  "metadata": {
    "source": "来源",
    "created_at": "创建时间",
    "updated_at": "更新时间",
    "version": "版本",
    "author": "作者",
    "status": "状态（active/inactive）"
  },
  "embedding": "向量（用于语义检索）"
}
```

### 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | string | 是 | UUID，唯一标识 |
| title | string | 是 | 知识条目标题 |
| content | string | 是 | 知识内容正文 |
| category | string | 是 | 主分类（业务知识/分析知识/操作知识/历史Case） |
| subcategory | string | 否 | 子分类 |
| tags | array | 否 | 标签数组 |
| metadata.source | string | 否 | 来源 |
| metadata.created_at | string | 是 | 创建时间（ISO格式） |
| metadata.updated_at | string | 是 | 更新时间（ISO格式） |
| metadata.version | string | 是 | 版本号 |
| metadata.author | string | 否 | 作者 |
| metadata.status | string | 是 | 状态（active/inactive） |
| embedding | array | 否 | 向量（检索时生成） |

## 三、维护机制

### 知识录入

- **录入方式**：
  - 手动录入：产品经理/运营人员手动添加知识条目
  - 自动沉淀：从对话历史中自动沉淀优秀案例
  - 批量导入：从文档中批量导入知识
- **录入流程**：
  1. 知识提交 → 2. 内容审核 → 3. 向量化处理 → 4. 上线生效
- **审核机制**：
  - 业务知识：需要业务负责人审核
  - 分析知识：需要产品经理审核
  - 操作知识：需要运营人员审核
  - 历史Case：自动沉淀，无需审核

### 知识更新

- **更新触发**：
  - 业务规则变更时更新
  - 用户反馈问题后更新
  - 定期复审优化
- **更新流程**：
  1. 提出更新 → 2. 内容审核 → 3. 版本升级 → 4. 重新向量化 → 5. 上线生效
- **版本管理**：
  - 使用语义化版本号（v1.0.0）
  - 保留历史版本，支持回滚
  - 记录更新日志

### 知识退役

- **退役条件**：
  - 业务下线
  - 规则变更
  - 内容过时
  - 被标记为inactive超过30天
- **退役流程**：
  1. 提出退役 → 2. 审核确认 → 3. 标记inactive → 4. 定期清理

## 四、调用规则

### 什么时候调用知识库

- **场景1**：用户询问Agent能力、使用方法
  - 查询操作知识库中的"Agent使用指南"和"FAQ"
- **场景2**：用户询问业务背景、数据集信息
  - 查询业务知识库中的"数据集映射表"和"业务背景"
- **场景3**：用户询问异动判断规则、分析口径
  - 查询分析知识库中的"异动判断规则"和"口径说明"
- **场景4**：生成分析结论时
  - 参考历史Case库中的"优秀分析案例"

### 检索策略

- **相似度阈值**：0.75（低于阈值不召回）
- **返回条数**：Top 5（按相似度排序）
- **重排序策略**：BM25 + 向量相似度融合

### 知识注入 Prompt 方式

- **方式1**：上下文注入（Context Injection）
  - 将检索到的知识插入到Prompt的上下文中
  - 适用于需要参考具体知识的场景
- **方式2**：系统提示词注入（System Prompt）
  - 将知识库内容写入系统提示词
  - 适用于需要遵循固定规则的场景

## 五、沉淀规则

### 从对话中沉淀

- **沉淀条件**：
  - 用户明确表示满意（"分析结论很准确"）
  - 报告被多次查看（>10次）
  - 被标记为优秀案例
- **沉淀流程**：
  1. 触发沉淀 → 2. 内容提取 → 3. 人工审核 → 4. 入知识库
- **审核机制**：
  - 需要产品经理审核
  - 确认分析结论的准确性和价值

### 从 Badcase 中沉淀

- **触发条件**：
  - 用户反馈分析结论错误
  - 机评指标低于阈值
  - 人评发现问题
- **沉淀流程**：
  1. Badcase记录 → 2. 原因分析 → 3. 修正方案 → 4. 入知识库
- **审核机制**：
  - 需要数据分析师+产品经理双重审核
  - 确认修正方案的正确性
