# 体验异动分析报告生成

## 架构说明

本 Skill 分两层执行。你（LLM）只负责第二层（文字解读），第一层（数学计算）由代码工具完成。

**不要自己做任何数学运算，直接读取计算工具输出的 JSON，基于其中的数字写结论。**

---

## 第一层：调用计算工具（anomaly_calc）

在你输出任何内容之前，必须先调用 `anomaly_calc` 工具函数：

### 输入参数

- `current_data`：本期两期明细数据（来自 friday-mcp-query）
- `compare_data`：对比期明细数据
- `daily_current` / `daily_compare`：日粒度数据（用于趋势图）
- `dimension_availability`：维度可用性清单

### 工具将自动完成

- 整体万服 YoY 计算
- 各维度逐项计算 delta / YoY% / 贡献度% / 次/万单贡献
- 异动打标（`up_major` / `down_major` / `new_added` / `disappeared` / `new_actual` / `extreme_value`）
- 极端值判断：上期<10 且本期>100 → 标 `new_actual`（不触发口径告警）
- 极端值判断：|YoY|>500% 且上期≥10 → 标 `extreme_value`（触发疑似口径告警）
- Top5 推高/压低排序（按贡献度%降序）
- 日趋势数组生成（7日，含本期+对比期）
- 各维度明细汇总表数据（按波动贡献排序）

## 第二层：你（LLM）的工作范围

接收 `anomaly_calc` 输出的 JSON 后，按以下模块顺序生成完整报告：

---

### 模块一：核心指标面板（必须输出）

**格式要求**：

```
【核心指标】
- 本期人工万服：{overall.current}（{本期周次} {本期日期范围}）
- 去年同期万服：{overall.compare}（{对比期周次} {对比期日期范围}）
- 万服同比：{上升/下降} {overall.yoy}%（差 {overall.delta}）
- 服务量（同比参考）：{overall.service_cnt}（+{overall.service_yoy}%，+{服务量增量}次）
- 订单量（同比参考）：{overall.order_cnt}（{overall.order_yoy}%）
```

**规则**：
- 万服同比 > 0 显示"上升"，< 0 显示"下降"
- 服务量增量 = 本期服务量 - 对比期服务量
- 保留两位小数

---

### 模块二：综合分析（必须输出）

#### 2.1 总结结论

**格式要求**：

```
【综合分析】

**总结**：{本期周次} {业务名}人工万服 {overall.current}，同比{上升/下降} {overall.yoy}%，主因为{服务量/订单量}同比{大幅增加/减少} {服务量同比}（{服务量增量}次），{订单量同比}，增速剪刀差{扩大/缩小}；{重点维度}为主要推高来源，{进线渠道}占比{XX}%，{重点FAQ}为最大单项FAQ。
```

**规则**：
- 若服务量增速 > 订单量增速 → 增速剪刀差扩大
- 若服务量增速 < 订单量增速 → 增速剪刀差缩小
- 重点维度：取 `dim.top_up[0]` 的维度类型
- 进线渠道占比：需从数据中提取最大进线渠道的服务量占比

#### 2.2 推高万服主因（红色标记）

**格式要求**：

```
**▲ 推高万服主因**
- {dim.top_up[0].name}（{dim.top_up[0].dim_type}）：{上升/下降} {dim.top_up[0].yoy}%（{增量}次）
- {dim.top_up[1].name}（{dim.top_up[1].dim_type}）：{上升/下降} {dim.top_up[1].yoy}%（{增量}次）
```

**规则**：
- 列出 Top2 推高因素（按贡献度排序）
- 增量 = `dim.top_up[n].curr_service - dim.top_up[n].prev_service`

#### 2.3 压低万服维度（绿色标记）

**格式要求**：

```
**▼ 压低万服维度**
- {dim.top_down[0].name}（{dim.top_down[0].dim_type}）：{下降/上升} {abs(dim.top_down[0].yoy)}%（{增量}次）
```

**规则**：
- 列出 Top1 压低因素
- 增量为负数时用"-"表示

#### 2.4 总量概览

**格式要求**：

```
**总量概览**
- 本期万服 {overall.current}，同比 {overall.yoy}%
- 本期服务量 {overall.service_cnt}，订单量 {overall.order_cnt}
- {服务量/订单量}增速 > {订单量/服务量}，是万服{上升/下降}主因
```

---

### 模块三：日万服趋势分析（必须输出）

#### 3.1 趋势图表描述

**格式要求**：

```
【日万服趋势】（本期 vs 去年同期，实线=本期，虚线=去年）

| 日期 | 本期万服 | 去年同期 | YoY% |
|------|---------|---------|------|
{遍历 daily_trend 数组生成表格行}
```

**示例**：
```
| 周一 3/16 | 12.68 | 9.52 | +33.2% |
| 周二 3/17 | 11.48 | 8.94 | +28.4% |
```

#### 3.2 趋势解读

**格式要求**：

```
**趋势解读**：本周万服呈现{上升/下降/波动}趋势，{周一至周五/周末}表现突出，{最高/最低}值出现在{日期}（{万服值}）。与去年同期相比，{整体/前半周/后半周}增长明显。
```

**规则**：
- 根据每日数据判断整体趋势
- 指出最高/最低值日期
- 分析与去年同期的对比差异

**📊 整体分析**

### 模块六：告警解读（必须输出）

对 JSON 中 `alerts` 数组里每条 alert，用一句人话说明含义和处置建议：

| 告警类型 | 说明格式 |
|---------|---------|
| `new_category` | "本期新增了{name}事件分类，可能导致数据虚高，建议排除后分析" |
| `near_zero_growth`（new_actual） | "{name}去年基数极小，本期出现实际业务，属正常增长，非口径问题" |
| `extreme_value` | "{name}变化幅度过大（YoY {yoy}%），疑似口径或分类调整，建议人工核查" |

---

### 模块七：不可用维度标注（条件输出）

对 `dimension_availability` 中 `available=false` 的维度，统一说明：

```
【数据说明】
- {维度名}维度当前数据集暂不支持查询，报告中该模块将展示数据缺失提示。
```

## 禁止行为（严格执行）

- ❌ 禁止自行计算任何数值（YoY%、贡献度%、绝对变化量等）
- ❌ 禁止修改 anomaly_calc 输出的数字
- ❌ 禁止对 available=false 的维度编造或推测数据
- ❌ 禁止将 new_actual 标签（近零增长）判断为口径异常
- ❌ 禁止省略任何 tag=up_major 或 down_major 的维度项
- ❌ 禁止遗漏任何已生成的模块
- ❌ 禁止在表格中使用 Markdown 加粗以外的格式（如颜色、背景色）
- ❌ 禁止在分析中加入未经计算工具验证的推测内容
